# 4.8 接口与数据结构设计（后端）

本文档对应《大创/结题材料》中 **4.8 接口与数据结构设计**、**4.9 系统部署说明**里与 **ForestFire 后端** 直接相关的部分，并与仓库实现保持一致。更细的接口示例见 `docs/前端须知.txt`、`docs/后端须知.txt`；环境与启动步骤见 `docs/环境.md`。

---

## 4.8.1 模块接口（应用内分层）

后端按「路由 → 业务/校验 → 持久化」组织，主要目录职责如下。

| 模块路径 | 职责 |
|----------|------|
| `app/api/` | HTTP 路由：认证、火点、仪表盘、台账、火焰分析上传 |
| `app/core/` | 鉴权依赖、JWT、统一业务异常 `ApiError` |
| `app/models/` | SQLAlchemy ORM 模型（MySQL 表映射） |
| `app/schemas/` | Pydantic 请求/响应校验与序列化 |
| `app/services/` | 模型推理等业务（如 `/api/analyze`） |
| `config/config.py` + `config/config.yaml` | 应用配置（数据库连接、JWT、上传目录等） |
| `main.py` | FastAPI 入口、生命周期、全局异常处理、轻量库表同步（见下文） |

**模块间调用关系（简述）**：`app/api/*` 通过 `Depends(get_db)` 取得 `Session`，读写 `app/models/*`；认证路由调用 `app/core/security` 做密码哈希与 JWT；分析路由调用 `app/services/model_service`。

---

## 4.8.2 系统间接口（对外 HTTP API）

**Base URL（开发）**：`http://localhost:8000`  
**API 前缀**：由 `config.yaml` 的 `api_v1_str` 决定，默认 **`/api`**。  
**统一响应**：`{ "code", "message", "data" }`，成功 **`code === 20000`**。  
**鉴权**：除注册、登录、根路径、`POST /api/analyze` 外，下列接口需在 Header 携带 `Authorization: Bearer <token>`。

### 接口一览（与实现一致）

| 方法 | 路径 | 说明 | 鉴权 |
|------|------|------|------|
| POST | `/api/auth/register` | 用户注册 | 否 |
| POST | `/api/auth/login` | 登录，返回 `access_token` | 否 |
| GET | `/api/auth/me` | 当前用户 | 是 |
| POST | `/api/analyze` | 图片/视频火焰分析（multipart） | 否 |
| GET | `/api/fire-markers` | 当前用户火点分页列表 | 是 |
| POST | `/api/fire-markers` | 创建火点 | 是 |
| PATCH | `/api/fire-markers/{id}` | 部分更新（含 `status/level/cause` 等；三者任一变化会写台账事件） | 是 |
| PATCH | `/api/fire-markers/{id}/status` | 仅更新状态；写台账事件 | 是 |
| DELETE | `/api/fire-markers/{id}` | 删除本人火点 | 是 |
| GET | `/api/fire-dashboard` | 仪表盘统计（全局聚合） | 是 |
| GET | `/api/fire-ledger` | 火情处置台账（事件表分页） | 是 |
| GET | `/api/geo/place-search` | 地名检索（后端代理高德 Web 服务） | 是 |
| GET | `/api/geo/reverse-geocode` | 逆地理（区县级 district/region） | 是 |
| GET | `/api/geo/map-config` | 返回 JSAPI key + security_js_code（供前端初始化地图） | 是 |

**OpenAPI**：服务启动后访问 `http://localhost:8000/api/openapi.json`（前缀以配置为准）。

## 4.8.3 消息字段（请求/响应约定）

### 4.8.3.1 统一 Envelope（所有 JSON 接口）

| 字段 | 类型 | 说明 |
|------|------|------|
| `code` | int | 业务码；成功 `20000` |
| `message` | string | 人类可读说明 |
| `data` | object \| null | 成功为业务数据；失败常为 `null` |

### 4.8.3.2 认证相关（节选）

- **登录响应 `data`**：`access_token`、`expires_in`、`user` 等（详见 `docs/前端对接-用户与火点标记.md`）。
- **请求头**：`Authorization: Bearer <access_token>`。

### 4.8.3.3 火点 FireMarker（核心枚举）

| 字段 | 取值 | 前端展示（见产品文案） |
|------|------|------------------------|
| `status` | `pending` / `handling` / `extinguished` | 未处置 / 处置中 / 已扑灭 |
| `level` | `low` / `medium` / `high` | 低 / 中 / 高 |
| `cause` | `human` / `lightning` / `farming` / `unknown` | 人为用火 / 雷击火 / 农事用火 / 其他未知原因 |

**分页查询**：`GET /api/fire-markers` 与 `GET /api/fire-ledger` 均支持 `page`、`page_size`（当前实现单页上限 **200**，与前端大图列表拉取对齐）。

## 4.8.4 数据库存储设计（MySQL）

数据库为 **MySQL 8.x**（推荐），连接串配置在 `config/config.yaml` 的 `database_url`。

### 4.8.4.1 表：`users`

存储注册用户；`password_hash` 为 bcrypt 哈希，不明文存密码。

### 4.8.4.2 表：`fire_markers`

火点主表：经纬度、标记时间、火焰处数、来源、备注，以及状态/等级/原因/地区、上报人冗余字段等。

**实现说明（与《后端须知》DDL 的差异，以代码为准）**：

- 归属字段：实现中使用 **`user_id`（VARCHAR(36)，外键关联 `users.id`）** 表示「该条火点归属用户」，功能上对应文档中的「标记人」语义。
- **`reporter_user_id`**：实现中为 **`VARCHAR(36)`**，与 `users.id`（UUID 字符串）一致，便于直接关联；若结题材料需写成 BIGINT，可在文档中说明「产品字段名」与「实现类型」的映射关系。
- 首次启动时：`main.py` 中 `Base.metadata.create_all()` 负责**新建表**；对已存在的旧库，另有**启动时补列**逻辑，避免仅改模型不迁移导致查询失败（生产环境建议逐步改为 Alembic 等正式迁移工具）。

### 4.8.4.3 表：`fire_marker_events`

火情台账事件表：`marker_id` 关联 `fire_markers.id`，记录某次处置相关展示字段及 `event_time`（与主表 `updated_at` 对齐写入）。

**业务规则（实现）**：

- `PATCH /api/fire-markers/{id}/status`：每次更新状态写入一条事件。
- `PATCH /api/fire-markers/{id}`：当 `status`、`level`、`cause` 任一相对原值发生变化时写入一条事件。
- 若主表 `region` 为空，事件表 `region` 实现中使用占位 **`未知`**，以满足 NOT NULL 与前端列表展示。

# 4.9 系统部署说明（后端）

本节只写 **后端部署**；前端、端侧、仿真/实机联调部署由对应团队文档说明。

## 4.9.1 部署环境

- **操作系统**：Windows / Linux / macOS 均可。
- **Python**：3.9+（见 `docs/环境.md`）。
- **数据库**：MySQL 8.x；先执行 `scripts/init_mysql.sql` 建库。
- **依赖**：`pip install -r requirements.txt`（版本以仓库 `requirements.txt` 为准）。

## 4.9.2 配置与密钥

1. 编辑 `config/config.yaml`：`database_url`、`jwt_secret`、`jwt_expire_seconds`、`environment` 等。  
2. **生产环境**务必更换强随机 `jwt_secret`，并限制配置文件访问权限。  
3. 模型权重：按 `docs/环境.md` 放置检测用权重文件（文件名以该文档为准）。

## 4.9.3 启动命令

```bash
python main.py
```

或使用 uvicorn（等价）：

```bash
uvicorn main:app --host 0.0.0.0 --port 8000
```

## 4.9.4 验证与联调

- 健康检查：`GET http://localhost:8000/` 应返回 `code: 20000`。  
- 接口文档：`GET http://localhost:8000/api/openapi.json`。  
- 业务联调清单：注册 → 登录 → `GET /api/auth/me` → `POST/GET` 火点 → 仪表盘/台账。

## 4.9.5 运维注意

- **日志**：关注 MySQL 连接失败、JWT 校验失败、以及 `/api/analyze` 推理异常。  
- **依赖**：`passlib` 与 `bcrypt` 主版本需匹配（见 `requirements.txt` 注释），避免启动期无意义告警。  
- **数据迁移**：当前仓库含启动时「补列」兜底；正式上线建议引入 Alembic 等迁移方案并关闭自动 DDL。
