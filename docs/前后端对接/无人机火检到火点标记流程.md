# 无人机火检（WebSocket）→ 火点标记（HTTP）全流程说明（给前端）

本文说明：**从收到 `uav.detection` 到成功 `POST /api/fire-markers` 创建火点**的后端行为、字段契约与常见失败原因。便于与 `persistFireMarkerFromDetection` 等逻辑对照排查。

---

## 1. 总流程（时序）

```
rosbridge 发布 /uav/state/fire_detection
        ↓
后端 WS 连接（已订阅该 topic）收到消息
        ↓
组装 payload，向浏览器 WebSocket 推送 { type: "uav.detection", payload }
        ↓
前端校验（去重、坐标合法等）
        ↓
POST /api/fire-markers  + Authorization: Bearer <access_token>
        ↓
后端 FireMarkerCreate 校验通过 → 写库 + 台账事件 → { code: 20000, data: ... }
```

- **火检不经由 HTTP 下发**：只有 **WebSocket** `uav.detection`。
- **落库火点只走 HTTP**：`POST {API_V1_STR}/fire-markers`（默认 **`/api/fire-markers`**，与 `config.yaml` 中 `api_v1_str` 一致）。

---

## 2. WebSocket：连接与消息类型

- **URL**：`ws(s)://<host>:<port>/ws/uav?token=<JWT access_token>`  
  - 路径为 **`/ws/uav`**（无额外 `/api` 前缀）。
- **鉴权**：Query `token` 与 HTTP 使用同一套 JWT。
- **服务端推送**（节选）：
  - `uav.telemetry`：位置/姿态等
  - `uav.detection`：**本条文档核心**
  - `fleet.telemetry` / `fleet.arrived`：车队相关
  - `ping`：心跳

前端需在收到 **`type === "uav.detection"`** 时进入火点创建逻辑。

---

## 3. `uav.detection` 的 `payload` 字段说明

后端在 `app/api/uav_routes.py` 中由 `/uav/state/fire_detection` 映射而来。

### 3.1 坐标（创建火点必填）

| 字段 | 类型 | 说明 |
|------|------|------|
| `latitude` | number | 优先使用 ROS 消息中的 `latitude`；若无则用当前 UAV 遥测里的纬度。 |
| `longitude` | number | 同上。 |

**重要**：若 ROS 与当前遥测都**没有**有效经纬度，后端**不会**推送 `uav.detection`（避免前端拿到空坐标反复失败）。请在前端仍做 `null`/范围校验。

### 3.2 机载原始字段（便于展示/调试）

| 字段 | 说明 |
|------|------|
| `fire_probability` | ROS 原样 |
| `risk_level` | ROS 原样（多为 0～1 浮点） |
| `flame_count` | ROS 原样 |
| `average_confidence` | ROS 原样 |
| `detected_target_count` | ROS 原样 |
| `fire_cause` | 已规范为 `human` / `lightning` / `farming` / `unknown` 之一 |
| `ts` | 后端生成 UTC ISO8601 时间字符串 |

### 3.3 与 `POST /api/fire-markers` 对齐的推荐字段（**建议直接用于请求体**）

为减少「字段名不一致导致 400」的问题，payload 中**额外**附带与 `FireMarkerCreate` 一致的键：

| 字段 | 类型 | 说明 |
|------|------|------|
| **`fire_count`** | int，≥1 | 由 `flame_count` / `detected_target_count` 推导，缺省为 1。 |
| **`level`** | `"low"` \| `"medium"` \| `"high"` | 由 **`risk_level`** 推导：数值 ≥0.67→`high`，≥0.34→`medium`，否则 `low`；若 `risk_level` 已是上述三字符串则原样使用。 |
| **`cause`** | 同上 `fire_cause` 枚举 | 与 `fire_cause` 一致，对应建库字段 **`cause`**。 |

创建火点时请使用：

- `latitude` / `longitude` ← payload  
- **`fire_count`** ← 请用 **`fire_count`**，不要用 `flame_count` 作为 POST 字段名（后端不认识 `flame_count`）。  
- **`level`** ← 请用 **`level`** 字符串，不要把未转换的 **`risk_level` 浮点**直接当 `level` 提交。  
- **`cause`** ← 请用 **`cause`**，与 `POST` 体一致（不要用 `fire_cause` 作为 JSON 键名除非你们自行映射）。

其余可选字段见下一节。

---

## 4. HTTP：`POST /api/fire-markers`

### 4.1 请求

- **方法**：`POST`
- **URL**：`{API_BASE}/fire-markers`  
  - 若 `API_BASE` 已含 `/api`（如 `http://localhost:8000/api`），则为 **`/api/fire-markers`**。  
  - **不要**重复拼成 `/api/api/fire-markers`。
- **Header**  
  - `Authorization: Bearer <access_token>`（必填）  
  - `Content-Type: application/json`

### 4.2 请求体（`FireMarkerCreate`）

**必填**

| 字段 | 要求 |
|------|------|
| `longitude` | float，-180～180 |
| `latitude` | float，-90～90 |
| `fire_count` | int **≥ 1** |

**常用可选（有默认值）**

| 字段 | 默认 | 约束 |
|------|------|------|
| `status` | `pending` | `pending` \| `handling` \| `extinguished` |
| `level` | `low` | `low` \| `medium` \| `high` |
| `cause` | `unknown` | `human` \| `lightning` \| `farming` \| `unknown` |
| `marked_at` | 服务端当前 UTC | ISO8601 |
| `source` | 无 | 最长 32，可填 `uav` 等 |
| `note` | 无 | 字符串 |
| `region` | 无 | 可不传；服务端可按高德逆地理写入 |

### 4.3 成功响应

```json
{ "code": 20000, "message": "成功", "data": { ...火点对象... } }
```

### 4.4 常见失败（为何「收到火检但库中没有火点」）

| 现象 | 常见原因 |
|------|----------|
| HTTP **400**，`code: 40000` | **请求体验证失败**（见 `message` 内字段提示）。典型：用了 `flame_count` 而不是 `fire_count`；`level` 传了数字或未在 `low/medium/high`；`cause` 不在四枚举内。 |
| HTTP **401**，`code: 40100` | 未带 Bearer、token 过期或无效。 |
| 从未收到 `uav.detection` | rosbridge 未连上、未订阅到 `fire_detection`、或后端因**缺经纬度**主动跳过推送（见 3.1）。 |
| 收到 WS 但不想发 HTTP | 属前端策略（去重、仅展示等），与后端无关。 |

校验失败时后端返回 **HTTP 400**（不是 422），body 形如：`{ "code": 40000, "message": "字段提示...", "data": null }`。请在 `persistFireMarkerFromDetection` 失败分支打日志或 toast 该 `message`。

---

## 5. 与 mock 联调时的注意点

- mock 仅在航点数量 **≥ 6** 时才会发 `fire_detection`（进而才有 `uav.detection`）；短航线（如返航 5 点）**不会有火检**。
- mock 在 `fire_detection` 里会带 `latitude`/`longitude`，后端会优先用于 `uav.detection`，一般不会出现「缺坐标不推」的情况。

---

## 6. 最小可用示例（前端 → 后端）

**WebSocket 收到的 `payload`（节选）**

```json
{
  "latitude": 40.029,
  "longitude": 116.255,
  "fire_count": 1,
  "level": "low",
  "cause": "unknown",
  "flame_count": 1,
  "risk_level": 0.22,
  "fire_cause": "unknown"
}
```

**据此构造 POST 体（示例）**

```json
{
  "latitude": 40.029,
  "longitude": 116.255,
  "fire_count": 1,
  "level": "low",
  "cause": "unknown",
  "status": "pending",
  "source": "uav"
}
```

## 7. 维护

- WebSocket 映射代码：`app/api/uav_routes.py`（`/uav/state/fire_detection` → `uav.detection`）。
- 建库接口：`app/api/fire_markers_routes.py`（`POST ""`）。
- 请求体模型：`app/schemas/fire_marker.py`（`FireMarkerCreate`）。

若后续机载 `risk_level` 语义与 0～1 分段不一致，可再约定分段规则或改为机载直接传 `level` 字符串并由后端透传。
